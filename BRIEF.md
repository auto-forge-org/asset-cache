# asset-cache Project Specification

## Executive Summary  
asset-cache is a high-performance digital asset management system designed for media companies, game studios, and design teams. It provides efficient storage, versioning, and retrieval of large media files with metadata-driven search capabilities. The system prioritizes scalability, security, and fast access through intelligent caching. It serves content creators, developers, and DevOps engineers who need reliable asset management with audit trails and access controls.

---

## Core Features  

1. **Asset Upload & Storage**  
   - Uploads binary assets with metadata  
   - Inputs: File stream (multipart/form-data), metadata JSON  
   - Outputs: Unique asset ID, storage location, checksum  
   - Edge cases: Large file chunking (>5GB), duplicate detection, invalid file types  

2. **Version Control**  
   - Tracks asset revisions with semantic versioning  
   - Inputs: Asset ID, update delta or full replacement  
   - Outputs: Version history, diff between versions  
   - Edge cases: Concurrent edits, rollbacks, version retention policies  

3. **Metadata Management**  
   - Structured key-value metadata with schema validation  
   - Inputs: Asset ID, metadata schema definition (JSON Schema)  
   - Outputs: Filtered search results, metadata validation errors  
   - Edge cases: Schema migrations, invalid metadata formats  

4. **Access Control**  
   - Role-based permissions (viewer/editor/admin)  
   - Inputs: User ID, asset ID, requested action  
   - Outputs: Authorization decision, audit log entry  
   - Edge cases: Permission inheritance, bulk operations  

5. **Search & Discovery**  
   - Full-text and metadata-based search with pagination  
   - Inputs: Search query (Lucene syntax), filters, sort parameters  
   - Outputs: Ranked asset list, total count  
   - Edge cases: Fuzzy search, field-specific weighting  

6. **Caching Layer**  
   - Distributed in-memory cache with TTL and eviction policies  
   - Inputs: Asset ID, cache expiration time  
   - Outputs: Cached asset stream or cache miss  
   - Edge cases: Cache stampeding, cache warming  

---

## Architecture Overview  
3-tier architecture with:  
- **API Layer**: Gin-based REST gateway with middleware for auth, rate-limiting, and logging  
- **Service Layer**: Business logic for asset processing, versioning, and access control  
- **Storage Layer**: Hybrid storage (S3-compatible object store + Redis cache) with metadata in PostgreSQL  
- **Key Patterns**:  
  - CQRS for read/write separation  
  - Event sourcing for audit trails  
  - Circuit breaker pattern for storage failures  
  - Cache-aside pattern for Redis integration

## Data Models  

```go
// Asset represents a stored media file
type Asset struct {
    ID          string    `json:"id" db:"id"`
    Name        string    `json:"name" db:"name"`
    Path        string    `json:"path" db:"path"`
    Size        int64     `json:"size" db:"size"`
    Checksum    string    `json:"checksum" db:"checksum"`
    MetadataID  string    `json:"metadata_id" db:"metadata_id"`
    OwnerID     string    `json:"owner_id" db:"owner_id"`
    CreatedAt   time.Time `json:"created_at" db:"created_at"`
}

// Metadata holds structured asset attributes
type Metadata struct {
    ID          string            `json:"id" db:"id"`
    Schema      map[string]string `json:"schema" db:"schema"` // JSONB
    Values      map[string]any    `json:"values" db:"values"` // JSONB
    AssetID     string            `json:"asset_id" db:"asset_id"`
}

// Version tracks asset revisions
type Version struct {
    AssetID     string    `json:"asset_id" db:"asset_id"`
    Version     string    `json:"version" db:"version"`
    ParentID    *string   `json:"parent_id" db:"parent_id"`
    Delta       []byte    `json:"delta" db:"delta"` // Binary diff
    CreatedBy   string    `json:"created_by" db:"created_by"`
    Timestamp   time.Time `json:"timestamp" db:"timestamp"`
}

// User represents a system user
type User struct {
    ID          string    `json:"id" db:"id"`
    Username    string    `json:"username" db:"username"`
    Roles       []string  `json:"roles" db:"roles"` // JSON array
    LastLogin   time.Time `json:"last_login" db:"last_login"`
}
```

## API / Interface Design  

### REST Endpoints  
1. `POST /api/v1/assets`  
   - Upload new asset  
   - Request: multipart/form-data with `file` and `metadata` fields  
   - Response: `201 Created` with `{ "id": "abc123", "checksum": "sha256:..." }`  

2. `GET /api/v1/assets/{id}`  
   - Retrieve asset with metadata  
   - Response: `200 OK` with full Asset object  

3. `GET /api/v1/assets/search`  
   - Search assets by metadata/query  
   - Query params: `q`, `page`, `per_page`  
   - Response: Paginated list with total count  

4. `POST /api/v1/assets/{id}/versions`  
   - Create new version  
   - Request: `{"delta": "base64_diff"}`  
   - Response: `201 Created` with version tag  

### CLI Commands  
```bash
# Cache warming
asset-cache warm-cache --query "type:video" --limit 100

# Audit log export
asset-cache export-audit --since "2024-01-01" --format jsonl
```

  - Structured JSON logs with trace IDs

  - 85%+ unit test coverage  
  - Chaos testing for cache failures
